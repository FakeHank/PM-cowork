## 一、产品概述

### 1.1 产品定位

PMWork 是一款面向产品经理个人工作流的 AI 原生工具。它不是一个通用的 AI 助手，也不是一个传统的文档编辑器——它是一个理解产品经理工作方式的专属工作台。

核心解决的问题：**从混沌的想法碎片，到可交付的产品文档，再到可交互的功能原型**——这条路径上的每一步，PM 今天都在用割裂的工具拼凑完成。PMWork 将这个过程整合为一个连贯的工作流。

### 1.2 核心设计理念

**AI 是协作者，不是工具。** PMWork 中的 AI 不只是被动地等待指令。它理解项目上下文、记住历史决策、主动发现遗漏——像一个经验丰富但从不抢风头的搭档。

**从续写开始，也可以从零开始。** PM 的大多数工作不是创建新文档，而是在已有资料的基础上推进。PMWork 的默认假设是"你已经有一些东西了"，同时也完整支持从零开始构建全新功能的场景。

**最低摩擦，永远是赢家。** 信息输入应该像自言自语一样自然。任何要求用户先分类、先整理、先想清楚再输入的设计，都是错误的。

### 1.3 目标用户

核心目标人群：日常工作中需要进行产品文档撰写和原型设计的产品经理。无论经验深浅、负责战略还是执行，只要工作产出涉及 PRD 和功能原型，都是 PMWork 的目标用户。

暂不考虑：跨团队协作场景。MVP 聚焦个人工作流。

### 1.4 产品形态

独立客户端应用（Desktop first），拥有自己的界面和交互体系。

## 二、核心概念模型

PMWork 围绕三个核心模块构建：

| 模块 | 隐喻 | 核心功能 | 信息特征 |
|------|------|---------|---------|
| **Inbox** 灵感碎片 | 私人日记本 | 低摩擦地收集一切 | 无结构、时间流、原始形态 |
| **Workspace** 文档工坊 | 项目工作间 | 对话驱动产出文档 | 结构化、版本化、可迭代 |
| **Canvas** 原型画布 | 白板 / 画板 | 可交互前端原型 | 可视化、可操作、可验证 |

三者的关系：

- **Inbox** 是原料池。信息在这里被无压力地捕获，等待被使用。
- **Workspace** 是加工厂。信息在这里被结构化、被讨论、被打磨成可交付的文档。
- **Canvas** 是展示台。文档在这里变成可以看到、可以点击、可以感受的原型。

信息流动方向以"Inbox → Workspace → Canvas"为主。

## 三、Inbox — 灵感碎片

### 3.1 模块定位

Inbox 是一个极低摩擦的私人信息收集器。它像一本永远打开的日记本，接受 PM 脑子里冒出来的任何东西——不需要分类，不需要整理，不需要想清楚再写。

它的价值不在于"管理信息"，而在于"**让记录这个动作的成本趋近于零**"。只有成本足够低，PM 才会真正把那些一闪而过的想法记下来，而不是让它们消失在忙碌的一天里。

### 3.2 信息输入方式

PMWork 支持以下输入方式，覆盖 PM 日常中最常见的信息来源场景：

**`P0` 文字输入** — 最基础的方式。打开 Inbox，直接打字。支持富文本也支持纯文本，不做强制格式要求。可以是一句话（"用户反馈说导出太慢"），也可以是一大段思考。

**`P0` 文件拖入** — 支持常见文件类型（PDF、Word、Excel、Markdown 等）。文件以附件形式存储，AI 提取其中关键信息生成摘要卡片。

**`P1` 语音输入** — PM 常常在走路、通勤、散步时冒出想法。长按语音按钮，说完自动转录为文字。保留原始音频作为附件，转录文本作为主体内容。

**`P1` 图片与截图** — 拖入或粘贴即可。竞品截图、用户反馈截图、白板照片、微信聊天截图。AI 自动识别图中内容并生成文字摘要，但原始图片始终保留。

**`P1` 剪贴板粘贴** — 从任何地方复制的内容（网页文字、链接、代码片段），直接 Cmd+V 粘贴进来。系统自动识别内容类型并适配展示。

**`P1` 外部导入** — 支持导入会议录音的转录文本、邮件正文、聊天记录等。以"原始内容 + AI 摘要"的形式保存。

### 3.3 信息存储与展示

**`P0` 时间流（Timeline）为主轴。** Inbox 的信息以时间倒序排列，最新的在最上面。这是最自然、最低认知负担的组织方式——不需要用户思考"这条信息应该放在哪个分类下"。

**`P0` 每条信息是一张卡片。** 卡片包含：原始内容（文字/图片/文件/音频）、创建时间、关联状态（是否已被发送到某个 Workspace Version）。

**`P0` 搜索。** 支持全文搜索、按时间范围筛选、按内容类型筛选。

**`P1` AI 自动标签。** 系统基于内容自动打标签，如"竞品""用户反馈""产品想法""技术约束""数据发现"等。标签不是为了分类管理，而是为了后续检索和关联。用户可以修改或删除标签，但永远不强制要求用户打标签。

**`P1` 自然语言搜索。** 支持自然语言检索（"上个月关于支付的所有想法"），AI 理解语义而非仅匹配关键词。

**`P1` 按标签筛选。** 基于 AI 自动标签进行筛选和归组浏览。

### 3.4 信息的流出

Inbox 中的信息可以通过以下方式流向 Workspace：

**`P0` AI 主动检索。** 在 Workspace 中与 AI 对话时，AI 可以根据当前讨论的内容，主动从 Inbox 中检索相关的碎片信息。AI 会以建议的形式呈现（"我在你的 Inbox 中发现了 3 条可能相关的记录，要查看吗？"），由用户决定是否纳入当前工作。这是最自然的流出方式——不需要用户提前手动整理，AI 在需要的时候自动找到相关内容。

**`P1` 手动发送。** 用户选中一条或多条 Inbox 卡片，点击"发送到"，选择目标 Project + Version。信息会被复制到该 Version 的 `references/` 目录下，原始 Inbox 卡片标记为"已关联"但不删除。

**`P1` 批量整理。** 用户可以要求 AI 对 Inbox 做一次整理（"帮我看看最近一个月的 Inbox，哪些和搜索功能相关"），AI 筛选并归组后，用户一键发送到对应的 Workspace。

### 3.5 AI 能力

| 优先级 | AI 能力 | 说明 | 触发方式 |
|-------|--------|------|---------|
| P0 | 智能搜索 | 支持全文检索，在 Workspace 对话中主动从 Inbox 检索相关内容 | 用户搜索时 / Workspace 对话中自动 |
| P0 | 内容提取 | 从文件中提取文字信息 | 输入后自动执行 |
| P1 | 自动摘要 | 为每条输入生成一句话摘要 | 输入后自动执行 |
| P1 | 自动标签 | 基于内容和上下文打标签 | 输入后自动执行 |
| P1 | 相关性聚类 | 发现内容相关的多条碎片，建议归组 | 定期自动执行 / 手动触发 |
| P1 | Project 关联建议 | 检测到某条碎片与已有 Project 高度相关时，建议关联 | 输入后自动执行 |

## 四、Workspace — 文档工坊

### 4.1 模块定位

Workspace 是 PM 的核心工作区。它不是一个文档编辑器——它是一个**以对话驱动、以版本管理为骨架、以上下文感知为基础的文档产出环境**。

PM 在这里与 AI 进行多轮对话，逐步将碎片化的想法和需求打磨成一份结构完整、逻辑自洽、可交付给开发团队的产品文档。

### 4.2 核心概念：Project 与 Version

#### 4.2.1 Project

Project 由用户自行创建和定义。它可以对应产品中的一个功能模块（如"搜索功能"），也可以对应一个业务领域（如"支付系统"），或者任何用户认为值得作为独立单元管理的产品方向。PMWork 不预设 Project 的粒度和命名方式——由用户根据自己的工作习惯决定。

一个 Project 下包含多个 Version，反映该方向从 v1 到 vN 的演进历史。Project 是长期存在的容器，只要这个方向还在迭代，Project 就一直在。

#### 4.2.2 Version

Version 对应一次具体的产品迭代。它是 PM 的基本工作单元。

每个 Version 只有两种状态：**进行中（Active）** 和 **已完成（Done）**。

这个状态的唯一作用是帮助界面做展示分层：Active 的 Version 在 Project 列表中优先展示，Done 的 Version 收纳进"更多"中，减少视觉噪音。状态由用户手动切换。

#### 4.2.3 Version 的文件结构

```
{project_name}/
  └── {version}/
      ├── spec.md                # 最终产出：产品需求文档
      ├── sessions/              # 与 AI 的对话记录
      │   ├── 2026-01-15-01.md
      │   ├── 2026-01-16-01.md
      │   └── ...
      ├── assets/                # 过程中收集的素材
      │   ├── competitor-screenshot-01.png
      │   ├── user-feedback-data.csv
      │   └── ...
      └── references/            # 上下文信息
          ├── prev-version-spec.md       # 前序版本的 spec
          ├── prev-version-summary.md    # AI 生成的前序版本摘要
          ├── inbox-fragment-01.md       # 从 Inbox 导入的碎片
          └── ...
```

**spec.md** — 唯一的核心产出物。所有对话、素材、参考信息，最终都是为了产出这一份文档。

**sessions/** — 保存每次与 AI 对话的完整记录。这不只是为了回溯，也是 AI 理解"这个 Version 经历了怎样的思考过程"的重要上下文。

**assets/** — 工作过程中用户主动上传或 AI 搜索找到的素材。竞品截图、数据表格、技术文档、设计参考图等。

**references/** — 当前 Version 的上下文依赖。最重要的来源是前序 Version 的信息，也包括从 Inbox 导入的碎片、从其他 Project 引用的文档等。

### 4.3 Version 的创建与继承

#### 4.3.1 创建新 Version

用户创建新 Version 时有两种选择：

**全新创建。** references 目录为空（或仅包含用户手动添加的内容）。适用于全新功能模块的首个版本。

**基于已有 Version 创建。** 用户选择一个"基准 Version"（通常是同 Project 下的前一个版本，但也可以跨 Project 选择）。系统将基准 Version 的以下内容带入新 Version 的 references：

| 继承内容 | 形式 | 说明 |
|---------|------|------|
| Project 大背景 | 原文 / AI 整理 | Project 级别的产品背景、目标、Roadmap 等长期不变的上下文信息，确保每个 Version 都理解"这个 Project 整体在做什么、往哪走" |
| 前序 spec.md | 原文复制 | 作为 `references/prev-version-spec.md` 保留完整原文 |
| 版本摘要 | AI 生成 | AI 基于前序 Version 的 spec 和 sessions 生成一份结构化摘要，包含核心功能点、关键决策、已知限制 |
| 遗留问题 | AI 提取 | AI 从前序 sessions 中提取标记为"待处理""下个版本再看"的问题 |
| 关键 assets | 用户选择 | 弹出前序 Version 的 assets 列表，用户勾选需要带入的素材 |

#### 4.3.2 继承的设计原则

- **完整的旧 spec 始终保留。** AI 摘要是辅助，不是替代。PM 经常需要回查"上个版本具体是怎么写的"。
- **AI 摘要侧重于"变化"和"决策"。** 不只是功能列表，而是"为什么这样做""放弃了什么""哪些是妥协"。
- **用户对继承内容有完全的控制权。** 系统提供默认建议，用户可以增减。

### 4.4 核心工作流：对话驱动文档

#### 4.4.1 进入 Version 的工作状态

每次用户打开一个 Version，AI 的第一步是**建立上下文感知**：

1. 扫描 references 目录，理解项目历史和当前依赖
2. 读取 spec.md 当前状态，识别已完成和未完成的部分
3. 回顾 sessions 中的历史对话，理解思考脉络
4. 检查 assets 中的素材，了解可用的参考信息

然后向用户呈现一个简洁的**上下文摘要**：

> "搜索功能 v2.1：spec 已完成 背景和目标 章节，用户故事写了 3 条（标记了 2 个 TODO）。上次对话讨论了筛选条件的优先级但未形成结论。references 中有 v2.0 的完整 spec 和 2 条来自 Inbox 的用户反馈。"

用户可以直接开始对话，不需要手动提供上下文。

#### 4.4.2 对话模式

用户与 AI 的对话是 Workspace 的核心交互。对话不是简单的 Q&A，而是一个逐步推进文档产出的过程。

**自由讨论。** 用户可以和 AI 自由地讨论问题，不必每句话都指向文档。比如"你觉得这个场景下用户会怎么想？""如果不做这个功能会怎样？"。AI 作为思维搭档参与讨论。

**指导 spec 更新。** 当对话产生了明确的结论，用户可以指示 AI 将结论写入 spec.md 的相应章节。AI 直接执行更新，并在对话中告知用户改了什么（"已更新 spec 的用户故事章节，新增了 2 条关于管理员角色的故事"）。

**AI 主动建议。** AI 在对话过程中可能发现需要补充的内容，会主动建议："你提到了管理员角色，但 spec 中的用户角色部分还没有描述管理员的权限，要现在补上吗？"

#### 4.4.3 Session 管理

每次对话自动保存为一个 session 文件（按日期编号）。

- 用户可以在一天内开启多个 session，也可以一个 session 跨越多天
- AI 在新 session 开始时会自动回顾之前的 sessions，保持上下文连贯
- 用户可以给 session 打标签或写备注（如"这次主要讨论了权限模型"），方便后续检索
- 用户可以要求 AI 为某个 session 生成摘要

#### 4.4.4 spec.md 的更新机制

AI 在对话过程中，当产生明确的结论或用户指示更新时，直接修改 spec.md 并在对话中告知用户本次变更的内容。例如：

> "已更新 spec：在 边界情况 章节新增了 3 条（用户未授权时的处理、网络中断时的降级、数据为空时的展示）；修改了 验收标准 第 2 条的措辞使其更具体。"

变更记录自动保存（包含变更内容、触发变更的 session、变更时间），用户可以随时查看历史变更或回退。

用户也可以直接编辑 spec.md（绕过 AI），系统会记录手动编辑并在下次对话时告知 AI。

### 4.5 spec.md 的文档结构

spec.md 采用结构化的 Markdown 格式。以下是推荐的标准模板，用户可以根据需要增减章节：

```markdown
# {功能名称} — 产品需求文档
> Project: {project_name} | Version: {version} | 状态: {Active/Done}
> 创建日期: YYYY-MM-DD | 最后更新: YYYY-MM-DD

## 1. 背景与问题
为什么要做这个功能？解决什么问题？面向谁？

## 2. 目标与度量
这个版本的核心目标是什么？如何衡量成功？

## 3. 用户故事
以 "作为{角色}，我希望{行为}，以便{目的}" 的格式描述核心场景。

## 4. 功能描述
详细的功能逻辑，包含核心流程和子流程。

## 5. 信息架构
页面结构、导航逻辑、数据展示层级。

## 6. 边界情况与异常处理
所有可能的边缘场景及其处理方式。

## 7. 验收标准
开发完成后可以逐条对照检查的具体标准。

## 8. 数据埋点
需要记录的用户行为和关键事件。

## 9. 灰度与发布策略
分批发布计划、回滚条件、实验设计。

## 10. 依赖与约束
技术依赖、其他功能依赖、已知限制。

## 11. 版本变更记录
相对上一版本的核心变更及原因。

## 附录
术语表、参考资料、开放问题。
```

### 4.6 AI 能力

#### 4.6.1 上下文感知

AI 在每次对话时都具备以下上下文：
- 当前 Version 的 spec.md 完整内容
- references 中的所有文档
- 该 Version 的历史 sessions
- assets 中的素材概览

AI 不需要用户反复解释"我们在做什么""上次聊到哪了"。

#### 4.6.2 断点续写

当 spec.md 处于半完成状态时，AI 能够识别哪些章节已完成、哪些为空或有 TODO 标记，并从断点处继续推进。续写时会保持与已有内容一致的风格和粒度。

#### 4.6.3 完整性检查

AI 对 spec.md 进行产品逻辑层面的完整性检查：

- **章节完整性**：是否有标准章节缺失
- **状态覆盖**：功能描述中是否覆盖了空态、错误态、loading 态
- **角色覆盖**：是否所有提到的用户角色都有对应的用户故事
- **验收标准**：是否足够具体到可以逐条验证
- **依赖标注**：是否识别了所有外部依赖

#### 4.6.4 一致性检查

- **内部一致性**：spec.md 内部是否存在自相矛盾的描述
- **跨版本一致性**：与 references 中的前序 spec 是否存在未说明的冲突
- **与对话的一致性**：spec.md 的内容是否反映了 sessions 中讨论的最终结论

#### 4.6.5 Reference 智能检索

在对话过程中，AI 会根据讨论的内容主动检索：
- references 中的相关段落（"上个版本是这样处理的…"）
- Inbox 中可能相关的碎片
- assets 中的相关素材

以建议形式呈现，不打断对话流。

## 五、Canvas — 原型画布

### 5.1 模块定位

Canvas 将 Workspace 中的文字描述变成可以在浏览器中点击、操作、感受的前端原型。

它的目标不是替代 Figma 或任何专业设计工具。它的定位是：**让 PM 在不依赖设计师的情况下，快速验证功能逻辑和信息架构是否正确。**

用 Tom Johnson 的话说——Canvas 产出的是一个"Bad Build"，它设计得不好看，但功能上是完整的。PM 的工作是在这个基础上挑刺、验证、迭代，然后把方向确认后的产物交给设计师去做真正的设计。

### 5.2 与 Workspace 的关系

每个 Canvas 实例绑定一个具体的 Project + Version。

**创建。** 用户在某个 Version 下点击"创建原型"，Canvas 读取该 Version 的 spec.md 作为生成原型的依据。

**同步。** 当 spec.md 更新后，Canvas 可以检测到变更并提示用户："spec 有更新（用户故事新增了 2 条），是否同步到原型？"用户确认后，AI 增量更新受影响的页面。

### 5.3 原型的定位与边界

#### 5.3.1 做什么

- **信息架构验证**：每个页面展示哪些信息、信息的层级和分组
- **交互流程验证**：页面之间的跳转、操作触发的状态变化
- **状态完整性**：每个页面的多种状态——正常态、空态、错误态、loading 态、首次使用态
- **核心文案**：按钮文字、提示文案、错误提示等关键微文案
- **响应式布局**：基本的布局结构（不追求像素级精确）

#### 5.3.2 不做什么

- **视觉设计**：不追求配色、图标、字体等视觉层面的精致度
- **高保真还原**：不试图达到 Figma 稿的还原度
- **动效与微交互**：不做复杂的过渡动画、手势交互
- **设计系统集成**：不对接已有的设计 token 或组件库

#### 5.3.3 原型的视觉风格

统一使用简洁的线框风格（wireframe），配色限制在灰度 + 一个强调色。这是有意为之：**让看到原型的人不会把注意力放在 UI 上，而是聚焦于功能和流程。**

### 5.4 核心工作流

#### 5.4.1 原型生成

用户在 Version 下发起创建原型。AI 读取 spec.md，按以下步骤生成：

1. **解析 spec**：提取功能描述、用户故事、交互流程、页面结构
2. **生成页面列表**：基于 spec 推断出需要多少个页面，每个页面的职责
3. **生成页面内容**：每个页面的信息架构、组件布局、关键文案
4. **建立页面关系**：定义页面间的导航和跳转逻辑
5. **生成状态变体**：为关键页面生成多种状态（正常/空/错误/loading）
6. **输出可交互原型**：生成可在 PMWork 内直接预览和操作的原型

生成完成后，用户可以直接在 PMWork 内预览和操作原型。

#### 5.4.2 对话驱动迭代

MVP 阶段，PM 通过与 AI 对话来迭代原型，不需要额外的批注工具：

PM 在 Canvas 内直接和 AI 对话，描述需要修改的内容（"把导航从侧边栏改成顶部 tab""这个列表页加一个空态""登录流程少了一步邮箱验证"）。AI 基于 spec 和当前原型的上下文理解意图，直接修改原型并告知改了什么。

这和 Workspace 中对话驱动 spec 更新的模式一致——PM 的核心交互方式始终是对话，而不是在界面上点点画画。

#### 5.4.3 导出与分享

原型的导出形式：

- **在线预览链接**：生成一个可分享的 URL，团队成员可以在浏览器中直接体验
- **截图导出**：将每个页面的每个状态导出为图片，用于 Figma 导入或文档嵌入
- **结构化描述导出**：导出页面列表、交互流程图、信息架构描述，用于和设计师交接

### 5.5 AI 能力

| AI 能力 | 说明 |
|--------|------|
| Spec 解析与页面推断 | 从文字描述的 spec 中推断出需要的页面、组件和交互 |
| 状态变体生成 | 自动为每个页面生成多种状态，覆盖 PM 容易遗漏的边缘态 |
| 对话意图理解 | 理解 PM 的自然语言描述并转化为具体的原型修改 |
| 一致性维护 | 修改某个页面时，自动检查是否影响其他相关页面 |
| Spec 差异检测 | 当 spec.md 更新时，识别变更影响的范围并建议原型更新 |
| 交互流程图生成 | 自动生成页面间的跳转关系图，帮助 PM 俯瞰整体流程 |

## 七、信息架构总览

### 7.1 核心页面布局：三栏结构

PMWork 的核心页面采用三栏布局：

```
┌─────────┬──────────────────────────┬─────────────────────┐
│ 左侧栏   │ 中间栏                    │ 右侧栏               │
│ (可收起)  │                          │                     │
│         │                          │                     │
│ + New   │                          │  文件夹浏览           │
│         │                          │  ├── spec.md        │
│ Inbox   │                          │  ├── sessions/      │
│         │      对话流                │  ├── assets/        │
│ Workspace│                          │  └── references/    │
│  ├ Proj A│                          │                     │
│  ├ Proj B│                          │  ─────────────────  │
│         │                          │                     │
│ Canvas  │                          │  文件预览             │
│         │                          │  (只读，点击文件时    │
│ ─────── │                          │   展示其内容)         │
│ Recents │                          │                     │
│         │                          │                     │
└─────────┴──────────────────────────┴─────────────────────┘
```

**左侧栏（导航栏，可收起）：** 全局模块导航。包含 New Chat（新建对话）、Inbox、Workspace（展开后显示 Project 列表，Active 版本优先，Done 版本收入"更多"）、Canvas、Recents（最近的对话记录）。类似 Claude 的侧边导航栏。

**中间栏（对话流）：** 核心工作区域。无论在 Inbox、Workspace 还是 Canvas 中，用户与 AI 的对话都在这里进行。这是用户花最多时间的地方。

**右侧栏（文件夹与文件预览）：** 当用户处于某个 Workspace Version 中时，右侧栏展示该 Version 的文件夹结构（spec.md、sessions/、assets/、references/）。点击某个文件时，在右侧栏内展示文件内容的只读预览。MVP 阶段不提供文件编辑能力——所有对文件的修改都通过中间栏的对话来驱动。

### 7.2 不同模块下的三栏表现

**Inbox 模式：** 中间栏展示时间流卡片 + 对话输入；右侧栏可以为空或展示选中卡片的详情。

**Workspace 模式：** 中间栏展示与 AI 的对话流；右侧栏展示当前 Version 的文件夹和文件预览。用户在对话中讨论 spec，同时可以在右侧随时查看 spec 当前内容和相关参考资料。

**Canvas 模式：** 中间栏展示原型预览 + 与 AI 的对话；右侧栏展示关联 Version 的 spec 和文件。

## 八、后续迭代方向

**近期（v1.1 - v1.2）**

- 竞品雷达模块：结构化跟踪竞品动态，与 Inbox 和 Workspace 联动
- 决策日志模块：在 spec 迭代过程中沉淀决策记录，支持检索和复盘
- Spec 模板自定义：用户定义自己的 spec 结构，替代默认模板

**中期（v2.0）**

- 多人协作：支持评审、批注、审批流（从个人工具扩展为团队工具）
- 外部工具集成：对接 Jira/Linear（需求状态同步）、Figma（设计稿引用）、数据平台（指标查看）
- Spec 到技术方案的桥接：辅助生成技术评审文档的初稿

**远期**

- 产品知识图谱：跨所有 Project 和 Version 构建功能关系图，理解整个产品的全貌
- 数据驱动的复盘：功能上线后接入数据，与 spec 中的目标和度量自动对比
- AI 产品教练：基于 PM 的工作模式给出个性化建议
